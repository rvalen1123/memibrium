## 1. Purpose

Lock the success criteria, slice definition, and decision rules for the next
gated append vs. query-expansion comparison **before** the run executes.
Prevents post-hoc threshold drift and gives the result a clean accept / reject
outcome.

The 25Q canary (2026-04-28) showed +2.0pp accuracy at +747ms latency. That
signal is too thin to promote on its own. This pre-registration governs the
next, larger run that will decide whether gated append moves from research to
a candidate for default-on.

## 2. Hypotheses

**H1 (primary, accuracy):** On a same-model slice ≥ 50Q from LOCOMO conv-26,
gated append yields strictly higher accuracy than the query-expansion-only
baseline.

**H2 (secondary, latency):** The latency cost of gated append vs. query-
expansion baseline does not grow superlinearly with question count.
Specifically, the per-question latency delta on the larger slice is within
1.5× the per-question delta observed on the 25Q canary (~30ms/Q).

**H3 (safety, no regression):** Gated append does not convert any fully-
correct baseline answer into a worse bucket (harmed-from-fully-correct = 0).

## 3. Slice definition

- **Source:** LOCOMO conv-26.
- **Minimum slice:** first 50Q (deterministic order, same seed/order as 25Q canary).
- **Preferred slice:** full conv-26.
- **Same-model constraint:** identical model, identical decoding parameters,
  identical retrieval config (other than the gated append toggle) across both arms.
- **Baseline must be re-run on the same slice.** The existing 25Q baseline
  (`locomo_conv26_query_expansion_25q_2026-04-28.json`) is not valid as a
  comparator for a 50Q+ run.

## 4. Pre-registered decision rules

A run promotes gated append from research to default-on candidate **only if
all of the following hold simultaneously**:

| Criterion | Threshold | Rationale |
|---|---|---|
| Accuracy delta (gated − baseline) | ≥ **+1.5pp absolute** on full slice | Above noise floor for n≥50; below this is canary territory |
| Harmed-from-fully-correct | **= 0** | Hard safety constraint; any regression here is disqualifying |
| Per-question latency delta | ≤ **1.5× the 25Q canary delta** (~45ms/Q ceiling) | Confirms no superlinear cost growth |
| Rescued count | **> harmed count** (already enforced by H3, but logged) | Net positive on contested questions |
| Slice size | n ≥ **50** | Below 50 stays research-only regardless of other numbers |

If accuracy delta is positive but below +1.5pp, the result is logged as
"directional, inconclusive" and a larger slice is required before any further
decision. **No promotion on partial criteria.**

## 5. Disqualifiers (run is invalid, not just negative)

- Baseline arm not re-run on the same slice.
- Background scoring flush loop active during either arm
  (`ENABLE_BACKGROUND_SCORING` must be `false` and verified end-to-end, not
  just by unit test).
- Different model, decoding params, or retrieval config between arms.
- Server restart between arms without a warm-up pass.
- Any test failure in `test_locomo_query_expansion.py`,
  `test_hybrid_retrieval_ruvector.py`, or `test_temporal_memory.py` on the
  commit used for the run.

## 7. Artifacts to produce

Naming follows existing convention.

- `docs/eval/results/locomo_conv26_query_expansion_<N>q_<DATE>.json`
- `docs/eval/results/locomo_conv26_gated_append_<N>q_<DATE>.json`
- `docs/eval/results/locomo_gated_append_<N>q_comparison_<DATE>.json`
- `docs/eval/results/locomo_gated_append_<N>q_comparison_<DATE>.md`

Comparison markdown must include:

- Slice size and source
- Per-arm accuracy, avg latency, p50, p95
- Bucket breakdown (unchanged-correct, same-partial, rescued,
  unchanged-wrong, harmed-from-fully-correct)
- Per-question latency delta (ms/Q) and ratio vs. 25Q canary
- Explicit promote / hold / reject verdict against the rules in §4
- Reference to this pre-registration file by filename

## 8. Reporting integrity

- Report all five buckets even if zero.
- Report latency as both average and p95. Average alone hides tail behavior,
  which is what blocks production at 8.98s.
- Do not retroactively re-define buckets, slice, or thresholds. If the rules
  in §4 turn out to be wrong, document the lesson in a follow-up
  pre-registration; do not edit this file's thresholds after the run.
